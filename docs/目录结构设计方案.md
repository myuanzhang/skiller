# skiller 目录结构设计方案

> 目的：把当前散乱的项目根目录整理成一个清晰、可长期维护的二次开发底座。
> 核心约束来自 `CLAUDE.md`：**简单优先、外科手术式修改**——不做超出需要的抽象。

## 一、现状盘点

```
skiller/                              ← 你的 git 仓库 (origin 已设, main 分支)
├── .git/                             ← skiller 自己的版本历史
├── AGENTS.md                         ← 指针文件 → CLAUDE.md + Superpowers 工作流约定
├── CLAUDE.md                         ← 编码行为准则 (单一真源 SSoT)
├── PRD - skiller.md                  ← 产品需求 (背景 + 目标)
├── Agent Skill 作用域隔离调研报告.md   ← 调研: scope 隔离 / .agents/skills SSoT
├── Superpowers Skill 手册（TRAE CLI 版）.md ← 调研: 开发工作流参考
└── skills-manager/                   ← 上游开源项目 (克隆自 xingkongliang/skills-manager)
    ├── .git/                         ← ⚠️ 嵌套的独立 git 仓库 (v1.25.2, 活跃维护)
    ├── src/ (React+TS 前端)
    ├── src-tauri/ (Rust 后端 + CLI)
    ├── docs/ scripts/ public/ assets/ skills/
    └── package.json 等配置
```

### 三个待解决的问题

| # | 问题 | 影响 |
|---|---|---|
| P1 | `skills-manager/` 带**嵌套 `.git`**，且未被 skiller 仓库跟踪 | 无法正常纳入版本管理；`git add` 会当成 submodule 或忽略内部文件 |
| P2 | 根目录 **文档与代码混放**（PRD/调研/准则散落在根） | 后续 app 代码进来后根目录会很乱，找不到重点 |
| P3 | 缺少根级 `.gitignore`、`README` | 构建产物 (`node_modules/`、`target/`、`dist/`) 会污染仓库 |

---

## 二、核心决策：如何处理上游代码（fork 策略）

这是最关键、且会影响后续所有工作的决策。三种方案：

### 方案 A — 吸收到根目录（Absorb / 扁平化）
删除 `skills-manager/.git`，把其内容上移到 skiller 根，代码与 skiller 仓库合一。

- ✅ 单一仓库，最扁平，符合"skiller 就是 skills-manager 的改名版"直觉
- ❌ **丢失上游历史**，日后无法 `git merge` 拉取上游更新（上游仍在活跃迭代 v1.25.x）
- ❌ 根目录代码 + 治理文档混在一起

### 方案 B — 上游作为 remote 的 fork（推荐）⭐
保留代码在子目录（改名为 `app/`），**保留其 git 历史**，把上游注册为 `upstream` remote，skiller 的定制提交在其之上。治理文档 + 调研归入 `docs/`，代码在 `app/`。

- ✅ **可持续同步上游**：`git -C app fetch upstream && git -C app merge upstream/main`
- ✅ 治理/文档（skiller 层）与 app 代码（fork 层）职责清晰
- ⚠️ 两层 git 仓库（skiller 管文档与顶层规划，app 管代码 fork），需明确各自职责边界
- ⚠️ 需要一次性厘清：app 是独立 fork 仓库，还是 skiller 的 submodule

### 方案 C — git subtree
把 skills-manager 以 `git subtree` 合入 skiller 单仓库，既单仓库又能拉上游。

- ✅ 单仓库 + 保留上游同步能力
- ❌ subtree 命令心智负担较高，solo 开发 ROI 低；与 CLAUDE.md「简单优先」冲突

**最终决策（已采纳）**：采用**方案 B 的目录结构（`docs/` + `app/`），但收敛为 skiller 单仓库**——删除 `app/` 的嵌套 `.git`，代码由 skiller 单一仓库直接跟踪，不做两层 git / submodule。详见 `decisions/0001-fork-策略.md`。放弃 `git merge upstream` 自动同步的便利，换取单仓库的简单性；未来如需上游更新，用临时 `git subtree` 或人工 cherry-pick。

## 三、推荐目录结构（已采纳：单仓库）

```
skiller/
├── .git/
├── .gitignore                      ← 新增: 忽略 node_modules/ target/ dist/ 等
├── README.md                       ← 新增: 项目总览 + 快速上手 (指向 docs/)
├── AGENTS.md                       ← 保留: Agent 行为入口 (指针)
├── CLAUDE.md                       ← 保留: 编码准则 SSoT
│
├── docs/                           ← 新增: 所有非代码文档归拢
│   ├── PRD.md                      ← 移动自「PRD - skiller.md」
│   ├── 目录结构设计方案.md           ← 本文档
│   ├── research/                   ← 调研资料
│   │   ├── Agent-Skill-作用域隔离调研.md
│   │   └── Superpowers-Skill-手册.md
│   └── decisions/                  ← 新增(可选): 架构决策记录 (ADR)
│       └── 0001-fork-策略.md        ← 记录本次 fork 方案决策
│
└── app/                            ← skills-manager 改名而来 (已去除嵌套 .git，由 skiller 单仓库跟踪)
    ├── src/ src-tauri/ ...
    └── package.json
```

### 命名理由
- `docs/`、`app/` 是通用、零歧义的约定，比 `skills-manager/` 更能表达职责。
- 中文文档保留中文名但去掉空格与全角括号（`Agent-Skill-作用域隔离调研.md`），避免脚本/CLI 处理路径时踩坑。
- 顶层只留最高频入口：`README` / `AGENTS` / `CLAUDE`。

## 四、执行步骤（已完成）

> 遵循 CLAUDE.md「外科手术式修改」：每一步可验证、可回滚。

```
1. 建目录 docs/ docs/research/ docs/decisions/                      ✅
2. 移动文档到 docs/ (原文档未被 git 跟踪，用 mv):                     ✅
   - PRD - skiller.md            → docs/PRD.md
   - Agent Skill 作用域隔离调研报告.md → docs/research/Agent-Skill-作用域隔离调研.md
   - Superpowers Skill 手册…      → docs/research/Superpowers-Skill-手册.md
   - 目录结构设计方案.md            → docs/目录结构设计方案.md (本文档)
3. skills-manager → app/ 并删除嵌套 .git (收敛单仓库):               ✅
   mv skills-manager app && rm -rf app/.git
4. 新增根级 .gitignore + README.md                                  ✅
5. 记录 ADR: docs/decisions/0001-fork-策略.md                        ✅
6. git add + commit (待用户授权后执行)                               ⏳
```

## 五、后续二次开发建议（在结构就绪之后）

来自对 skills-manager 架构的分析，与你已知偏好对齐：

1. **品牌化改造**：`com.agentskills.desktop` identifier、产品名、图标、`tauri.conf.json` 里的 updater endpoint（否则会拉上游 release）。这些改动集中在 `app/src-tauri/tauri.conf.json` 与前端文案。
2. **路径策略本土化**：上游认为 Codex 只读 `~/.codex/skills`；而你的 SSoT 实践用 `~/.agents/skills/` 软链。需在 `app/src-tauri/src/core/tool_adapters.rs` 调整 adapter 路径以匹配你的约定。
3. **scope 隔离**：结合 `docs/research/Agent-Skill-作用域隔离调研.md`，在 project workspace 逻辑中落实项目专属 skill 的防跨项目误触发。
4. **上游同步节奏**：单仓库放弃了自动 merge；如需上游更新用临时 `git subtree` 或人工 cherry-pick（详见 `decisions/0001-fork-策略.md`）。

> 以上第 5 节为方向性建议，非本次结构整理的执行内容；具体实现待单独规划。
