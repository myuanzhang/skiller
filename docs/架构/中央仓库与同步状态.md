# 中央仓库与同步状态

> 本文说明「技能中心」（即**中央仓库**）在 skiller 中的作用，以及 Agent 页面
> 「本地 Skills」列表上各同步状态标签（仅本地 / 已同步 / …）的含义与判定逻辑。

## 一句话模型

「技能中心」= 中央仓库 = 所有 Skill 的**单一数据源（Single Source of Truth）**。
各 Agent 目录里看到的 Skill，本质上都是从中央仓库**部署/链接过去的副本**。

- 磁盘位置：默认 `~/.skiller`，可在设置页「中央仓库路径」自定义。
- 每个 Skill 的正本（central copy）都在这里，数据库存有对应的受管理记录
  （`SkillRecord`，含 `central_path` 与内容哈希）。

## 中央仓库的四个作用

### 1. 集中存储 + 统一管理
- 所有 Skill 正本集中在中央库，DB 记录里有 `central_path` 和内容哈希。
- 「技能中心」页面是它的可视化界面：浏览全部 Skill、批量打标签、按来源/标签
  过滤、查看文档与版本历史。

### 2. 向各 Agent 分发（一处管理、多处生效）
- 从中央库把 Skill「同步」到某个 Agent，默认用**符号链接**（symlink），
  占用空间极小、改动实时生效。
- 一个 Skill 改一次，链接到它的所有 Agent 同时更新，无需逐个复制。
- 各 Agent 本地副本会持续与中央库版本做哈希比对，得出下面的同步状态。

### 3. 配合 Preset 做「技能组」编排
- Preset（预设）本质是「中央库里哪些 Skill 组合起来启用」的清单。
- 删 Preset 不会删中央库里的 Skill —— Skill 归属始终在中央库，Preset 只是引用。

### 4. 作为 Git 备份 / 多机同步的单元
- 开启 Git 备份后，备份与同步的就是**整个中央仓库**。
- 换设备或多机协作时，只要同步中央库，所有 Skill、标签、Preset 关联都随之带过去。
- 注意：项目里的符号链接指向**本机**中央库，不会随 Git 仓库带走 —— 链接是本机
  部署产物，正本才是中央库。

**印证：** 从中央库删一个 Skill，会连带移除「中央仓库文件、Preset 关联、
以及已同步到各 Agent 的目标」（`app/src/i18n/zh.json:109`）。

## Agent 页面的同步状态

Agent 页面「本地 Skills」列表上的状态标签，反映**该 Agent 本地副本**与
**中央仓库版本**的对比结果。代码里叫 `sync_status`。

- 判定逻辑：`app/src-tauri/src/commands/projects.rs:455`（`classify_sync_status`）
- 文案定义：`app/src/i18n/zh.json:901`（`globalWorkspace.localSkills.status`）
- 前端渲染：`app/src/views/WorkspaceView.tsx:182`（`getLocalStatusMeta`）

一共 **5 种**状态：

| 状态 key | 文案 | 含义 | 颜色 |
|---|---|---|---|
| `localOnly` (`project_only`) | 仅本地 | 中心没有，只存在于该 Agent 本地（卡片有「上传」入口可纳入中心） | 灰色 |
| `inSync` (`in_sync`) | 已同步 | 本地副本与中心版本内容一致（哈希相同） | 绿色 |
| `project_newer` | 本地有变更 | 本地副本比中心新（本地被改过） | 琥珀色 |
| `center_newer` | 中心有变更 | 中心版本比本地新（可拉取更新） | 天蓝色 |
| `diverged` | 冲突 | 两边都改过、无法判断新旧，需手动处理 | 紫色 |

### 判定顺序

1. 先比内容哈希：一致 → **已同步**（`in_sync`）。
   - 快路径比 DB 里存的中心哈希；DB 哈希可能过期，再从磁盘实时重算中心哈希兜底。
2. 哈希不一致，再比修改时间（阈值 1 秒）：
   - 本地更新 → **本地有变更**（`project_newer`）
   - 中心更新 → **中心有变更**（`center_newer`）
   - 时间戳无法区分 → **冲突**（`diverged`）
3. 中心无对应受管理记录 → **仅本地**（`project_only` / `localOnly`）。

### 相关操作
- `center_newer`（中心有变更）和 `diverged`（冲突）状态下，界面提供「拉取」
  按钮，用中心版本覆盖本地（`app/src/views/WorkspaceView.tsx:545`）。
- `localOnly`（仅本地）状态下，卡片提供「上传」入口，把本地 Skill 纳入中央库管理。

## Agent 详情页的数字汇总

进入某个 Agent 详情页，标题下会看到一行汇总，如：

```
Claude Code  [42]
~/.claude/skills · 42 / 1 个受管理 / 1 个已同步
```

这些数字全部基于该 Agent **本地 skills 目录**扫描到的 Skill（`localSkills`），
文案 key 为 `globalWorkspace.localSkills.summary`。

| 显示 | 变量 | 含义 |
|---|---|---|
| `~/.claude/skills` | `currentTool.skills_dir` | 该 Agent 读取 Skill 的本地目录 |
| 标题徽章 + total（如 `42`） | `localSkills.length` | 该目录下扫描到的 Skill **总数**，**包括不是通过 Skiller 安装的** |
| `N 个受管理`（managed） | `managedLocalCount` | 总数中被**技能中心纳管**的数量：能在中央仓库找到对应记录（`center_skill_id` 且中央库存在）的 Skill（`app/src/views/WorkspaceView.tsx:414`） |
| `N 个已同步`（synced） | `inSyncLocalCount` | 受管理中 `sync_status === "in_sync"` 的数量：本地副本与中央库版本**内容完全一致**（`app/src/views/WorkspaceView.tsx:397`） |

### 三者的包含关系

```
总数  ⊇  受管理  ⊇  已同步
```

- **总数**：目录里的全部 Skill，大多是「仅本地」（手动放入、或其它工具安装，技能中心不认识）。
- **受管理**：其中有多少能和中央库正本对上号。
- **已同步**：受管理中有多少此刻与中央库内容一致（无本地改动、也不落后于中心）。

所以「42 / 1 个受管理 / 1 个已同步」= 目录有 42 个 Skill，仅 1 个被 Skiller 纳管，
且那 1 个当前是同步状态；其余 41 个是该 Agent 自有、Skiller 未接管的。受管理但
未同步的 Skill 会显示为上文的「本地有变更 / 中心有变更 / 冲突」，未纳管的显示「仅本地」。

### 侧边栏 Agent 数字 = 「受管理」数

侧边栏每个 Agent 名称后的徽章数字，对应详情页的 **「N 个受管理」**（managed），
既不是 total（42），也不是「已同步」。

计算方式（`app/src/components/Sidebar.tsx:76`，`globalSkillsByAgent`）：

```ts
map[tool.key] = managedSkills.filter((skill) =>
  skill.targets.some((target) => target.tool === tool.key)
).length;
```

即：从**中央仓库**统计「有多少个受管理 Skill 把这个 Agent 列为同步目标」。

这与详情页 managed 是**同一批 Skill 的两个方向**：
- 侧边栏：从中央仓库看「有几个 Skill 指向该 Agent」
- 详情页 managed：从 Agent 本地目录看「有几个 Skill 被中心纳管」

正常情况下两者相等。侧边栏不显示 total，是因为它只数技能中心管的 Skill，忽略
「仅本地」的那些；也不按「已同步」过滤，只要是指向该 Agent 的受管理 Skill 就计入。

## Agent 本地目录的断链识别

Agent 页面会在加载本地 Skills 时自动检查该 Agent 的 `skills_dir` 下是否存在
**断链符号链接**（broken symlink），不需要用户手动点击检测按钮。

### 什么会被识别为断链 Skill

后端命令 `detect_broken_symlinks` 只扫描 Agent 的 skills 根目录：

- 只检查 symlink 本身，不递归扫描子目录
- 相对 symlink 目标会先按 symlink 所在目录解析，再判断目标是否存在
- 目标不存在时返回 symlink 名称
- 跳过 `.` 开头的隐藏项，例如 `.ruff_cache`，因为这类通常是工具缓存，不是 Skill

例如 `~/.trae-cn/skills/AFFiNE -> /Users/bytedance/.skillshub/AFFiNE`，
若目标目录不存在，会被当作断链 Skill 返回；而
`~/.trae-cn/skills/.ruff_cache -> /Users/bytedance/.skillshub/.ruff_cache`
即使是断链，也不会显示在 Agent 页面。

### 前端显示与清理

`WorkspaceView` 会把正常扫描到的本地 Skill 与断链结果合并渲染：

- 正常 Skill 继续显示同步状态、标签、文件数和上传/拉取/删除操作
- 断链 Skill 会补成一张卡片，显示路径说明
- 断链卡片使用红色文本、红色边框和删除线标识
- 断链卡片不打开详情、不提供上传/拉取，只保留删除入口

删除断链时，后端会先确认请求路径位于该 Agent 的 `skills_dir` 内；如果该路径是
断链 symlink，则只删除 symlink 本身，不跟随也不删除它指向的目标路径。

## 技能中心卡片：Agent 图标与「启用」按钮

技能中心每张 Skill 卡片右侧有两组控件，它们是**互不相关的两套机制**，容易混淆。

### Agent 图标（SyncDots）——各 Agent 的安装状态

卡片会列出**所有已安装且启用的 Agent**，用明暗区分该 Skill 是否已同步到它，
而不是「只显示已安装该 Skill 的 Agent」。逻辑见
`app/src/components/SyncDots.tsx:61`。

| 外观 | 状态 key | 含义 |
|---|---|---|
| 图标清晰/高亮 | `synced` | 该 Skill **已同步安装**到这个 Agent（在 `skill.targets` 里） |
| 图标半透明（opacity 45%） | `available` | 该 Agent 已安装，但这个 Skill **尚未同步**过去 |
| 琥珀色边框 | `orphan` | 已同步，但该 Agent 当前不可用（被卸载/禁用）。仅详情面板显示（卡片未传 `includeOrphan`） |
| 末尾 `+N` | — | 图标最多显示 6 个（`limit={6}`），其余折叠为计数 |

**图标可点击**（`onToggle`，`app/src/views/MySkills.tsx:1789`）：
- 点半透明图标 → 把 Skill 同步安装到该 Agent（`syncSkillToTool`）
- 点高亮图标 → 从该 Agent 卸载（`unsyncSkillFromTool`）
- 悬停提示：「点击安装」/「点击卸载」

> 要点：图标代表「系统里所有可用 Agent + 各自的安装状态」，**只有高亮的才真正
> 装了这个 Skill**，半透明的是「可装但未装」。

### 「启用」按钮——加入当前 Preset，与 Agent 无关

「启用」按钮控制的是**当前选中的 Preset**（如 Default），不是 Agent
（`handleTogglePreset`，`app/src/views/MySkills.tsx:761`）：

- **「启用」** → 把 Skill **加入当前 Preset**（`addSkillToPreset`）。
- 加入后按钮变绿显示 **「已启用」**，再点则从 Preset 移除（`removeSkillFromPreset`）。

**它不会一键安装到所有 Agent。** 它只是把 Skill 归入某个技能组清单；真正决定
同步到哪些 Agent 的，是上面那排可点击图标，或详情面板里的逐个 Agent 开关。

**一句话对照：**
- 图标 = 「这个 Skill 在各 Agent 的安装状态」（亮=已装、暗=未装、可点切换）
- 启用 = 「把 Skill 加进当前 Preset 技能组」，与装到哪些 Agent 是两码事
