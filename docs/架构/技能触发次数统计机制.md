# 技能触发次数统计机制

技能卡片上的「N 次」角标显示每个 skill 被触发的**总**次数；hover 角标可看到
按 agent 拆分的次数（如 `trae: 39`）。数据由后端命令
`get_skill_usage_stats`（`commands/skills.rs`）在调用时**现场聚合**，前端
`MySkills.tsx` 启动时拉取，按 `skill.name` 索引后传给
`SkillListCard` / `SkillGridCard` 渲染角标（仅 `count > 0` 时显示）。

## 数据源

`collect_usage_stats()` 合并以下来源，产出 `SkillUsageStatDto { skill_name,
count, last_used_at, agents }`。其中 `agents` 是按 agent 拆分的
`SkillUsageAgentStatDto { agent, count, last_used_at }` 列表（按次数降序），
使总次数可下钻到每个 agent 各触发了几次；聚合逻辑见 `add_usage_event`。

1. **TRAE CLI session 日志**（主力来源）——
   `~/.trae/cli/sessions/**/*.jsonl`。每行一个事件；一次 skill 触发对应一行
   `type=response_item`、`payload.type=function_call`、`payload.name=="Skill"`，
   被触发的 skill 名来自 JSON 编码的 `payload.arguments.skill`，时间取该行
   顶层 `timestamp`（RFC3339），agent 记为 `"trae"`。解析见 `collect_usage_trae`
   / `trae_skill_call_name`。
2. `<中央库>/usage/*.jsonl` —— skiller 自有埋点格式（`collect_usage_jsonl`）。
3. `~/.claude/skill-stats.json` —— Claude Code 的统计文件（`collect_usage_json`）。
   注：本机实测该文件不存在、Claude Code 也不产生它，且其 transcript
   （`~/.claude/projects/**/*.jsonl`）里无 skill 触发信号；这条来源目前无数据，
   保留仅为兼容其他环境。

`skill.name` 由 `skill_metadata::infer_skill_name` 得出（frontmatter `name`，
否则回退目录名），与 TRAE `arguments.skill` 使用的 slug 天然对齐，因此角标能匹配。
TRAE 里出现、但中心未纳管的 skill 会被统计但无对应卡片，静默不显示（无害）。

## 关键决策：为什么解析 TRAE session，而不是让 agent 上报

skiller 是外部管理器，agent 运行时它并不「在场」，无法直接观测触发；解析 TRAE
session **零侵入**、能回溯历史，优于要求每个 agent 配合埋点上报的方案。决策全文
与权衡见 [`docs/adr/0001-skill-usage-from-trae-sessions.md`](../adr/0001-skill-usage-from-trae-sessions.md)。

## 计数口径

同一次 `/skill` slash 触发会同时产生一条 `<command-name>` 标记和一条 `Skill`
function_call。为避免双重计数，**只统计 `Skill` function_call**，它已覆盖手动
slash 与模型自动调用两类触发。
