# ADR 0001 — 上游代码处理策略：收敛为 skiller 单仓库

- 状态：已采纳
- 日期：2026-07-03

## 背景

skiller 基于开源项目 `xingkongliang/skills-manager` 二次开发。下载下来的
`skills-manager/` 带有独立的嵌套 `.git`（v1.25.2，上游活跃维护）。需要决定
如何在 skiller 仓库中安置这份上游代码。

曾评估三种方案：
- A 吸收扁平化（代码上移至根，删上游历史）
- B fork + upstream remote（保留上游 `.git`，两层仓库）
- C git subtree（单仓库 + 可同步）

## 决策

采用 **方案 B 的目录结构，但收敛为 skiller 单仓库**：

- 目录布局遵循方案 B：文档归 `docs/`，代码归 `app/`。
- 删除 `app/` 的嵌套 `.git`，所有代码由 skiller 单一仓库直接跟踪
  （不使用 submodule，也不保留两层 git）。

## 理由

- 单仓库最简单，符合 `CLAUDE.md`「简单优先」——solo 开发无需承担
  两层 git / submodule 的心智负担。
- `docs/` + `app/` 的职责分层仍然保留，根目录清爽。

## 影响与后续

- 放弃了「`git merge upstream/main` 自动同步」的便利。
- 上游仍活跃，未来如需拉取更新，方案是：临时注册上游 remote 并用
  `git subtree` 拉取到 `app/`，例如：

  ```bash
  git remote add upstream https://github.com/xingkongliang/skills-manager.git
  git fetch upstream
  # 首次将 app/ 认作 subtree 后，可用 subtree pull 合并上游更新
  git subtree pull --prefix=app upstream main --squash
  ```

  注：由于初始导入是直接 `mv` + 删 `.git`（非 subtree add），首次 subtree
  pull 可能需要手动处理合并基线；届时按需评估，必要时改为人工 cherry-pick
  关键上游修复。

- 品牌化改造（identifier / 产品名 / updater endpoint）会使 `app/` 与上游
  进一步分叉，届时更依赖人工挑拣上游变更，这是可接受的取舍。
